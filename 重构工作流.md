## 一、架构选型建议

### 推荐方案：Vite + 原生 JS 模块化

不需要 React/Vue。你的项目是**内容驱动型教学站**，原生 JS + ES Modules 完全够用，而且 Claude Code 生成原生 JS 的质量和可控性远高于框架代码。

### 为什么选 Vite？
- GitHub Pages 友好：`vite build` 生成纯静态文件
- 开发体验好：HMR 热更新，改 CSS/JS 即时生效
- ES Modules 原生支持：代码按需加载
- 零配置即可使用，不会给 Claude Code 增加复杂度

## 二、Claude Code 工作流（分 7 个阶段）

### 阶段 0：项目初始化

在 Claude Code 中使用 **plan 模式**：

```
/plan

我要将一个 6600 行的单文件教学网站重构为 Vite 多文件项目，部署到 GitHub Pages。

请读取以下文件了解项目全貌：
- CLAUDE.md（项目规范）
- design-spec.md（设计规范）
- todo.md（已完成功能清单）
- index.html（当前代码，6600行）

然后帮我：
1. 初始化 Vite 项目（vanilla JS，不用框架）
2. 建立上述文件夹结构
3. 从 index.html 提取 :root CSS 变量到 src/styles/variables.css
4. 建立路由系统（hash 路由，保持 #0 ~ #24 兼容）
5. 搭建侧边栏组件

不要急着迁移页面内容，先把骨架搭好。
```

执行后切回正常模式让它动手。


### 阶段 3：逐批迁移页面（每次 3-4 个页面）

这是最耗时的阶段，分批进行。每批给 Claude Code 的 prompt 模板：

```
请迁移 p3（色彩理论）、p4（色彩和谐）、p5（科研配色）到独立模块。

对于每个页面模块 (src/pages/pX-name.js)：
1. 导出一个 render() 函数，返回页面 HTML 字符串
2. 导出一个 init() 函数，负责绑定事件和初始化交互
3. 导出一个 cleanup() 函数（可选），页面离开时调用
4. 从 src/utils/color.js 导入需要的颜色工具函数
5. 从 src/components/ 导入需要的组件

HTML 内容从 index.html 对应的 <div class="page" id="pageX"> 中提取。
JS 逻辑从 index.html 的 initPage(X) case 和相关函数中提取。

不要简化或删减任何现有功能。确保所有交互完整迁移。
```

**建议迁移批次：**
- 第 1 批：p0（Hero）、p1、p2 — 最简单，验证架构
- 第 2 批：p3、p4、p5、p6 — 色彩核心模块
- 第 3 批：p7、p8 — 色彩进阶
- 第 4 批：p9、p10、p11、p12 — Illustrator + 导出
- 第 5 批：p13、p14、p15、p16 — AI 工作流
- 第 6 批：p17、p18、p19 — 图表核心（最复杂，单独处理）
- 第 7 批：p20、p21 — R 配色与出版
- 第 8 批：p22、p23、p24 — PPT 设计

每批完成后：
```
请运行 npm run dev 测试刚迁移的页面。
确认以下功能正常：
- 路由跳转到对应页面
- 所有交互组件可用
- CSS 样式正确

```

## 五、常见陷阱

1. **不要让 Claude Code 一次重写整个文件**
   - 6600 行→一次性重构 = 必定丢失功能
   - 始终分模块、分批次

2. **每完成一批就 git commit**
   ```
   每完成一个阶段后，请帮我：
   git add -A && git commit -m "阶段X: 具体描述"
   ```

3. **保留旧文件做对照**
   - 把原始 index.html 保留为 legacy/index-original.html
   - 迁移每个页面时可以对照检查功能是否遗漏

4. **测试优先**
   - 先迁移最简单的页面（p0, p1, p2）验证架构
   - 再迁移最复杂的页面（p17, p18, p19）验证极端情况
   - 中等复杂度的页面批量处理

5. **Claude Code 上下文窗口有限**
   - 不要在一次会话中让它处理超过 1000 行的新代码
   - 复杂页面（如 p18 的 16 种图表）分 2-3 次完成
